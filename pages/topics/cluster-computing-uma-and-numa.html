Overview

What you should know by the end of this lesson

Unit 2 expects you to compare different ways of organising processing and memory access. Some systems spread work across several whole computers, while others use several processors inside one shared-memory system. The key is to explain how the design affects performance, scaling, and suitability.

1 How many machines? Cluster questions are about several separate computers or nodes working together.
2 Is memory shared equally? UMA questions focus on one shared memory space with the same access time.
3 Does locality change speed? NUMA questions focus on local memory being faster than remote memory.
Cluster example

College VLE on results day

Many students refresh pages at once. Several web nodes can share those requests, and traffic can move away from a failed node.

Exam link: scalability and resilience across separate machines.

UMA example

Small shared-memory workstation

A workstation with a few processors can use one shared memory pool. Each processor gets the same route to the data, so the model is predictable.

Exam link: equal access time, simpler memory behaviour.

NUMA example

Large virtualisation host

A server may have several processor sockets, each with nearby memory. Keeping a virtual machine close to its local memory can reduce latency.

Exam link: local memory is faster than remote memory.

Quick memory hooks

  • Cluster: several separate computers or nodes work together.
  • UMA: each processor sees shared memory with the same access time.
  • NUMA: memory access time depends on where the data is located.
  • Choice: think about scaling, latency, and the kind of workload.

Where it appears

These ideas appear in architecture comparison questions, server-style scenarios, and longer answers where you must explain why one design scales better or handles memory access more effectively than another.

Cluster Computing

Cluster computing uses several separate machines to share work

A cluster is a group of computers, often called nodes, linked together so they can act as one larger system for a task or service. The work may be divided between nodes to improve throughput, reliability, or scalability.

Many nodes instead of one bigger machine Shared workload across the cluster Useful for scaling and resilience
Requests or jobs Work queue
Node 1 Separate computer
Node 2 Separate computer
Node 3 Separate computer

The cluster behaves like one service to the user, but the work is coordinated between multiple machines.

Scalability More nodes can be added when more performance or capacity is needed.
Redundancy If one node fails, the service may continue through the others.
Distributed work Tasks may be split across many machines rather than one shared memory space.
Management cost More machines mean more network coordination, monitoring, and system administration.

Practical context: busy online services

Imagine an online enrolment system during a deadline. One server may not handle every request on its own, and an outage would stop students from submitting forms. A cluster lets several nodes share the workload, while redundancy helps the service continue if one node fails.

UMA

Uniform memory access means each processor sees memory equally

In a UMA system, processors share the same main memory and the access time to that memory is treated as uniform. One processor does not have a faster route to one part of memory than another processor.

Processor 1 Processor 2 Processor 3
Same access time for each processor
Shared main memory One memory pool, equal route length
UMA feature What it means Implication
Shared memory pool Processors use one shared main memory space. Programming and memory behaviour can be easier to reason about.
Uniform access time Memory latency is the same whichever processor is asking. Performance is predictable, but scaling can become harder as more processors contend for the same shared resource.
Simpler coordination There is less need to think about local versus remote memory. Good for smaller shared-memory multiprocessor systems.

Practical context: smaller shared-memory systems

UMA can suit a smaller multiprocessor server or workstation where programs need shared memory but the processor count is not huge. For example, a media-editing workstation may benefit from several processors sharing the same memory pool without the software needing to place data near a particular socket.

Typical trade-off

UMA is simpler to describe and often simpler to manage, but as processor count rises the shared-memory path can become a limit. That makes it less attractive for some larger systems.

NUMA

NUMA keeps shared access but memory is not equally close to every processor

In a NUMA system, processors still work in one overall system, but some memory is local to a processor or socket and some is remote. Accessing nearby memory is faster than accessing memory attached elsewhere, so memory access is non-uniform.

Processor A
Local memory A Fast for Processor A
Remote access Still possible, but slower
Processor B
Local memory B Fast for Processor B

Local memory is faster

A processor reaches its nearest memory with lower latency than memory attached to another processor.

Better scaling

NUMA is often used when a larger shared-memory system needs to support more processors and more memory.

Locality matters

Performance is strongest when a task mainly uses memory close to the processor running it.

Practical context: large database or virtualisation servers

NUMA often appears in larger servers with multiple processor sockets. A database process or virtual machine will usually run better when its most-used data stays in memory attached to the same socket, because remote memory takes longer to reach.

Why NUMA can be harder to explain

NUMA is not just "faster UMA". The important difference is that memory latency changes depending on location. That can improve scalability, but it also means memory placement matters more.

Comparison

Choose the model that fits the workload and system size

Strong answers link the architecture model to the shape of the workload. The exam usually rewards the explanation of why the choice fits, not just the correct label.

Online shop sale

Cluster

Use several nodes behind the service so requests can be shared and the shop can keep running if one node fails.

Small workstation

UMA

Use a shared memory pool when equal access time and simpler programming are more important than very large scaling.

Large host server

NUMA

Use local memory near each processor socket so bigger shared-memory systems can scale with less contention.

Model Main idea Typical strength Possible drawback
Cluster computing Several separate nodes share the work. Scales out well and can improve resilience. Needs network coordination and more management.
UMA Processors share memory with equal access time. Simpler shared-memory behaviour. Shared access can limit scaling.
NUMA Processors share memory, but access time varies. Supports larger systems more effectively. Remote memory access is slower.

Scenario thinking

  • Choose cluster computing when the work can be spread across several separate machines or the service needs redundancy.
  • Choose UMA for a smaller shared-memory system where equal access and simpler memory behaviour matter.
  • Choose NUMA for a larger processor and memory system where better scaling is worth the extra complexity.

Exam Traps

Common mistakes students make with these architecture models

These topics often lose marks when answers blur together "multiple computers", "multiple processors", and "shared memory".

Calling a cluster one big shared-memory computer

A cluster is made from multiple separate nodes, not one single processor-memory design.

  • Clusters distribute work across separate machines.
  • UMA and NUMA describe memory access inside a system.

Fix: Ask whether the model is about many machines or shared memory.

Thinking NUMA means no memory is shared

NUMA still describes one overall system. The difference is that memory is closer to some processors than others.

  • Local access is faster.
  • Remote access is slower.

Fix: Use the phrase non-uniform access time.

Assuming the most complex model is always best

The exam usually wants suitability, not "most advanced" technology.

  • UMA can be the better fit for a smaller system.
  • Clusters are useful only when the workload suits them.

Fix: Link the model to the job the system must do.

4 marks Explain

Question 1

Explain the difference between UMA and NUMA in terms of memory access and state one implication of that difference.

Show answer guide

A strong answer defines both models and links the difference to performance or scaling.

  • In UMA, processors share memory with the same access time.
  • In NUMA, some memory is local and faster while remote memory is slower to access.
  • This means NUMA is non-uniform in latency.
  • One implication is that NUMA can scale better for larger systems, but memory placement matters more.
8 marks Recommend and justify

Question 2

Recommend a suitable architecture approach for a busy online service that must stay available, a smaller shared-memory workstation, and a large virtualisation server with several processor sockets. Justify each choice.

Show answer guide

The best answers justify each choice clearly.

  • A busy service spread across several machines is a good fit for cluster computing.
  • That is because multiple nodes can share the workload and improve resilience.
  • A smaller shared-memory server may be well suited to UMA.
  • UMA offers one shared memory pool with equal access time for each processor.
  • A large virtualisation server with several processor sockets may be well suited to NUMA.
  • NUMA allows each processor to use local memory more quickly, while still allowing remote memory access when needed.
  • The justification should explain why distributed nodes, equal access, or memory locality fit each scenario.

Glossary

Quick glossary

Cluster computing
An approach where several separate computers or nodes work together on a task or service.
UMA
Uniform memory access, where processors share memory with the same access time.
NUMA
Non-uniform memory access, where some memory is faster to reach than other memory.
Node
One individual computer within a cluster.

Remember

Separate machines is not the same as shared memory

  • Clusters are about several nodes working together.
  • UMA and NUMA are about memory access inside a system.
  • NUMA differs from UMA because locality changes speed.
