Overview

What you should know by the end of this lesson

Backup and recovery questions are really about risk and continuity. The system needs a plan for what happens when data is deleted, corrupted, lost, encrypted, or made unavailable.

Learning goals

  • Explain the purpose of backups clearly.
  • Describe how recovery planning reduces disruption.
  • Distinguish between resilience and true backup.
  • Apply backup ideas to realistic failure scenarios.

Quick memory hooks

  • Backup: an extra recoverable copy.
  • Recovery: getting data or service back.
  • Resilience: keeping things running if possible.
  • Continuity: reducing downtime.

Typical threats

Hardware failure, accidental deletion, malware, corruption, device theft, fire, and flooding can all make data unavailable. Backup planning exists because these things do happen.

Protection

Backups matter because important data can be lost in many different ways

A backup is an extra copy that can be used later if the live data becomes unusable. The key idea is recoverability, not just duplication.

Recovery

Recovery planning decides how quickly and how fully a system can return to normal

It is not enough to have backups somewhere. Organisations also need to think about how to restore data, who is responsible, how long it will take, and what should be restored first.

Decision 1

Backup frequency decides how much recent work could be lost

Frequent backups reduce the amount of new data lost after a failure, but they can cost more storage, bandwidth, processing time, and administration.

09:00 12:00 15:00 smaller gap

Exam phrase: choose frequency by how often the data changes and how much loss the organisation can tolerate.

Decision 2

Storage location decides whether the backup survives the same incident

Local backups may restore quickly, but off-site or cloud copies help if the main building, network, or storage device is damaged or stolen.

main site separate copy off-site

Exam phrase: avoid keeping every copy in the same physical or logical place.

Decision 3

Restore time affects downtime and business continuity

A backup strategy must consider how long restoration takes. Some data can wait, but critical services may need a faster recovery route.

minutes hours days impact grows

Exam phrase: a backup is less useful if restoring it takes too long for the organisation's needs.

Decision 4

Testing proves that recovery actually works

Organisations need to test restore procedures, not just assume that stored files are usable. Testing can reveal missing files, corrupt backups, poor instructions, or access problems.

restore sample check access verify data trusted plan

Exam phrase: untested backups may fail at the exact moment they are needed.

Decision 5

Recovery priorities decide what returns first

During a major incident, not everything can always be restored at once. A recovery plan should identify the most important data, users, services, and dependencies.

1 critical records 2 core systems 3 archive data

Exam phrase: prioritising recovery reduces disruption by restoring the most important services first.

Distinction

Resilience helps availability, but it is not the same as having a backup

Students often confuse features that reduce downtime with features that create a separate recoverable copy of data. Both matter, but they solve different problems.

Backup extra restorable copy RAID improved resilience Recovery plan to restore service

Resilience

Keeps service available through some failures.

Example: RAID tolerating a drive failure

Backup

Provides a separate copy that can be restored later.

Example: off-site copy from last night

Worked comparison

RAID can keep data available after a drive failure, but it does not protect against accidental deletion, ransomware, or a fire that destroys the whole system. A real backup strategy is still needed for those risks.

Exam Traps

Common mistakes with backup and recovery

Most lost marks come from treating any form of duplication as a full backup solution or from ignoring the recovery stage.

Assuming RAID is enough

RAID helps with some failures, but not all kinds of data loss.

Fix: Separate resilience from true backup.

Only discussing storage, not restore

A backup is not useful if the organisation cannot restore it quickly enough.

Fix: Mention recovery time, testing, and priorities.

Ignoring off-site risk

If the only copy is in the same building, a major incident may destroy both.

Fix: Consider location as well as frequency.

Question 2

A college stores important student records digitally. Discuss the factors that should affect its backup and recovery strategy.

Show answer guide
  • The college should consider how often records change and how much data loss is acceptable.
  • Restore time matters because long outages affect teaching and administration.

Glossary

Quick glossary

Backup
An extra copy kept so data can be restored later.
Recovery
The process of restoring data or service after failure.
Resilience
The ability to keep running through some kinds of failure.
Off-site backup
A backup stored away from the main location.

Remember

Protect and restore

  • Backups exist because failures and mistakes happen.
  • Recovery planning matters as much as storing the copy.
  • Resilience features do not replace proper backups.
