Overview

What you should know by the end of this lesson

An interrupt is a signal that makes the CPU pause its current work so something important can be dealt with. Strong Unit 2 answers explain not just that the CPU pauses, but how register values are protected so the interrupted task can continue properly afterward.

Learning goals

  • Define an interrupt clearly.
  • Explain why interrupts are useful in computer systems.
  • Describe how register contents are saved and restored.
  • Link interrupt handling to correct program execution.

Quick memory hooks

  • Interrupt: a request for immediate CPU attention.
  • Save state: keep the current register values safe.
  • Service routine: code that handles the interrupt.
  • Restore state: put the old register values back.

Where they appear

Interrupts are used when devices or events need attention while a program is already running, such as keyboard input, timers, communication events, or hardware signals.

Interrupts

An interrupt tells the CPU that something needs attention now

Instead of constantly checking every device in turn, the CPU can respond when an interrupt signal arrives. That makes the system more efficient because the processor can continue normal work until attention is genuinely needed.

Pause current work is temporarily suspended Respond the CPU handles the urgent event Resume the original task continues afterward
Why it matters The CPU can react quickly to important events without wasting time checking devices unnecessarily.
Typical example A key press, printer status update, timer signal, or incoming data event can trigger an interrupt.
Main advantage Interrupts improve responsiveness and make better use of CPU time.
Key implication The CPU must return to the interrupted task accurately once the interrupt has been handled.

Worked example

Imagine a user presses a key while a program is already running. The interrupt tells the CPU that keyboard input needs attention. The CPU pauses the current task briefly, handles the key press, and then continues the original task.

This is better than repeatedly polling the keyboard every moment even when no key has been pressed.

Register Handling

Registers must be saved and restored so work can continue correctly

When an interrupt happens, the CPU may already have important values stored in its registers. If those values were overwritten and lost, the interrupted program might resume incorrectly. That is why the current register state is saved before the interrupt is serviced and restored afterwards.

Step What happens Why it matters
1. Save register state The current values in relevant registers are stored safely. The interrupted task can be resumed exactly.
2. Handle the interrupt The CPU runs code that deals with the interrupt request. The urgent event is processed promptly.
3. Restore register state The saved register values are put back. The original program continues with the correct context.
4. Resume execution The CPU returns to the interrupted task. Normal processing carries on rather than starting over.

Program counter link

The CPU must remember where execution should continue once the interrupt has been handled.

Register link

Temporary working values in registers must be preserved so the interrupted calculation or process is not corrupted.

System link

Good interrupt handling keeps the system responsive without breaking the task that was already in progress.

Worked sequence example

Suppose the CPU is partway through a calculation and several register values are holding active data. A timer interrupt arrives. The CPU first saves the current register contents and the point it has reached in the program. It then runs the interrupt service routine for the timer event. When that is finished, the saved register values are restored and the original calculation continues from the correct point.

Comparison

Separate the interrupt itself from the handling process

Students often treat interrupts as if they are only a sudden stop. Stronger answers explain both the trigger and the recovery: why the CPU paused, how it protected the current state, and how it resumed correctly.

Question focus Best explanation Why it fits
Why interrupts exist They let the CPU react to events when attention is needed. This shows the efficiency and responsiveness benefit.
What happens to current work The current state is saved before the interrupt is handled. This explains how the original task is protected.
Why registers matter here Registers may contain active values that must be restored. This connects interrupts to register handling directly.
How execution continues The CPU restores the old state and resumes the interrupted program. This shows why the task can continue accurately.

Use interrupt language when...

The question asks why the CPU pauses or why the system can react quickly to events.

Use register-handling language when...

Exam reminder

The CPU does not just "go back later." It must save and restore the processing context properly.

Exam Traps

Common mistakes students make with interrupts

Interrupt questions often look straightforward, but weak answers usually miss the role of saved state and correct resumption.

Treating an interrupt like an error

An interrupt is not automatically a fault. It is simply a request for CPU attention.

  • A keyboard input can trigger an interrupt.
  • A timer signal can trigger an interrupt too.

Fix: Describe it as a signal or request, not always as a problem.

Forgetting the saved register state

Some answers say the CPU pauses and resumes, but never explain how it avoids losing the current values it was using.

  • Active register values may still be needed later.
  • Those values must be restored before normal work continues.

Fix: Include the words save, restore, or state when the question asks about handling.

Assuming the CPU restarts the program from the beginning

Good interrupt handling resumes from the correct point rather than starting the whole task again.

  • The original location in the program must be remembered.
  • The old register context must be restored first.

Fix: Explain that the CPU returns to the interrupted task, not to the start of the program.

Question 1

Explain why a keyboard interrupt is useful in a computer system instead of making the CPU check the keyboard continuously.

Show answer guide

A strong answer focuses on efficiency and timely response.

  • The CPU does not need to waste time checking the keyboard constantly.
  • The interrupt only requests attention when a key is actually pressed.
  • This lets the CPU keep doing other useful work in the meantime.
  • The system still responds quickly when input is needed.
5 marks Explain and justify

Question 2

Explain why register handling is important when an interrupt occurs, and what could go wrong if the current register state were not restored correctly afterwards.

Show answer guide

The best answers connect register handling to processing context and correct resumption.

  • The CPU may already be using registers to store active values and the current execution context.
  • Those values must be saved before the interrupt routine changes them.
  • After the interrupt is handled, the saved register values are restored.
  • This allows the interrupted program to continue from the correct point with the correct data.
  • If the state were not restored, the program could resume with the wrong values or wrong position and behave incorrectly.

Glossary

Quick glossary

Interrupt
A signal that asks the CPU to deal with something important.
Register state
The current values stored in registers while processing is happening.
Interrupt service routine
The code that handles the interrupt event.
Restore
Put the saved register values back so the original task can continue.

Remember

Pause, protect, handle, resume

  • An interrupt pauses the current work.
  • The current register state must be protected.
  • The CPU should resume the original task correctly afterwards.
