Overview

What you should know by the end of this lesson

Stored-program architecture explains how a computer can keep a program in memory and run it one instruction at a time. In Unit 2 you need to compare the Von Neumann and Harvard models clearly, including how their memory arrangement affects performance and suitability.

Learning goals

  • Define stored-program architecture clearly.
  • Describe the features of Von Neumann architecture.
  • Describe the features of Harvard architecture.
  • Explain the Von Neumann bottleneck and choose a suitable architecture for a scenario.

Quick memory hooks

  • Stored program: instructions are kept in memory, not fixed by wiring alone.
  • Von Neumann: instructions and data share the same main memory and pathway.
  • Harvard: instructions and data use separate memory paths.
  • Trade-off: simplicity vs reduced contention.

Where it appears

These ideas appear in architecture comparisons, processor performance questions, and scenario answers where you must justify why one computer design is more suitable than another.

Stored Program Model

Programs are stored in memory so one machine can run many jobs

A stored-program computer keeps instructions in memory in a form the CPU can fetch, decode, and execute. That means the same hardware can behave very differently depending on which program has been loaded.

Store the instructions in memory Fetch the next instruction into the CPU Run the program without rewiring the machine
Program loaded Machine-code instructions are placed in memory.

Idea Meaning Why it matters
Instructions are stored The program sits in memory as binary machine-code instructions. The CPU can fetch the next instruction during the instruction cycle.
Same hardware, different software The processor does not need to be rewired for each new task. A general-purpose computer can run a word processor, game, or browser on the same system.
Memory access becomes important The CPU must keep reaching memory for instructions and often for data as well. This is where the difference between Von Neumann and Harvard architecture becomes important.

Why the stored-program idea was powerful

Earlier machines could be much more awkward to change because the task was closely tied to the hardware setup. Stored-program architecture made computers flexible because loading a new program changed the behaviour of the machine.

Von Neumann

Von Neumann architecture uses shared access for instructions and data

In the Von Neumann model, program instructions and data share the same main memory. They also use the same pathway or bus between the CPU and memory, so instruction fetches and data transfers must take turns.

Shared design

Von Neumann layout

CPU Fetches instructions and works on data
One shared bus or pathway
Main memory Stores instructions and data together

The same route is used when the CPU fetches the next instruction and when it reads or writes data.

Instruction fetch CPU asks memory for the next instruction.
shared bus one route takes turns
Data access CPU also needs values from memory.
One memory space Instructions and data are held in the same main memory.
Shared pathway The CPU and memory communicate over the same route.
Simpler organisation This model is straightforward and flexible for general-purpose computing.
Von Neumann bottleneck Instructions and data compete for the same access path, which can slow the CPU.

Why the bottleneck happens

Imagine the CPU needs to fetch an instruction and then read a value from memory to carry it out. In a Von Neumann system, both actions use the same route, so they cannot happen over that path at exactly the same moment. That contention is the Von Neumann bottleneck.

Harvard

Harvard architecture separates instruction access from data access

In the Harvard model, instructions and data use separate memories or separate pathways. This means the CPU can fetch an instruction while also working with data, which helps reduce contention.

Because the routes are separate, instruction fetch and data access can happen without competing for the same path.

Instruction path Fetch program code from instruction memory.

Data path Read or write values in data memory.
Separate memories or paths Instructions and data are not forced through one shared route.
Reduced bottleneck The CPU can fetch instructions and access data more efficiently.
Good for dedicated systems This model often suits embedded controllers, microcontrollers, and signal-processing tasks.
Extra complexity Separate organisation can increase design complexity and may not be the simplest choice for every system.

Where Harvard-style ideas are useful

A controller that repeatedly checks sensors and updates outputs benefits when it can keep pulling the next instruction while also reading data values. That is why Harvard-style ideas are often linked to embedded and specialist systems.

Comparison

Choose the architecture that best fits the job

Strong exam answers do not just name the model. They connect the structure of the architecture to a real benefit or limitation in the scenario.

Choose Von Neumann when simplicity and flexibility matter

  • Shared memory for instructions and data
  • Simple stored-program model
  • Good fit for many general-purpose systems

Choose Harvard when separate access improves throughput

  • Separate instruction and data paths
  • Reduced contention during repeated processing
  • Good fit for specialist or embedded systems
Feature Von Neumann Harvard What it means in practice
Memory arrangement Instructions and data share memory. Instructions and data are separated. Harvard reduces contention, while Von Neumann keeps the design simpler.
Pathway or bus use One shared route Separate routes Harvard can fetch instructions and access data without competing for one path.
Typical strength Flexible, simple stored-program organisation Higher efficiency for repeated dedicated workloads The better choice depends on the purpose of the system.
Common drawback Von Neumann bottleneck Greater complexity Questions often ask you to trade simplicity against throughput.

General-purpose computer

In exam terms, a general-purpose desktop or laptop is usually matched to the Von Neumann model because one stored-program memory arrangement is simple and flexible.

Embedded controller

A dedicated controller may fit Harvard architecture better if it benefits from fast repeated instruction fetches alongside data access.

Real-world reminder

Many modern processors borrow ideas from both models, but exam questions usually want a clear comparison of the textbook Von Neumann and Harvard designs.

Exam Traps

Common mistakes students make with stored-program models

These questions usually lose marks when answers remember the names but forget the layout, the bottleneck, or the reason one model suits a scenario better.

Explaining stored program too vaguely

Some answers say only that the computer stores things in memory, which is too broad.

  • Say that program instructions are stored in memory.
  • Link that to the CPU fetching and executing those instructions.

Fix: Use the phrase instructions stored in memory.

Forgetting why the Von Neumann bottleneck happens

Students often name the bottleneck without describing the cause.

  • The problem is shared access to one memory route for both instructions and data.
  • The CPU may need both, so they compete for the same path.

Fix: Always mention the shared pathway or shared bus.

Saying Harvard is always better

Harvard reduces contention, but the exam still expects you to think about complexity and suitability.

  • A dedicated controller may benefit more than a simple general-purpose system.
  • Strong answers justify the choice instead of treating it as universally best.

Fix: Compare the model to the scenario, not to an imaginary perfect system.

4 marks Explain

Question 1

Explain why Von Neumann architecture is described as a stored-program architecture and include one effect of its shared memory access.

Show answer guide

A strong answer explains both the stored-program idea and the shared-access implication.

  • The program instructions are stored in memory in a form the CPU can fetch and execute.
  • This means the same hardware can run different programs without being rewired.
  • In Von Neumann architecture, instructions and data share the same memory route.
  • That can create a bottleneck because instruction fetches and data access compete for the same path.
6 marks Recommend and justify

Question 2

Recommend a suitable architecture model for a school computer room desktop and for a traffic-light controller. Justify both choices using the memory and data-access design of each model.

Show answer guide

The best answers justify both recommendations.

  • A school desktop is usually described with the Von Neumann model in Unit 2 because it is a simple, flexible, general-purpose stored-program design.
  • Its instructions and data share memory, which is a reasonable trade-off for a general-purpose computer.
  • A traffic-light controller is a strong fit for Harvard architecture because it is a dedicated embedded system.
  • Separate instruction and data access can reduce contention while the controller repeatedly checks sensor inputs and updates outputs.
  • The answer should make it clear that the choice depends on the purpose of the system, not just which model sounds more advanced.

Glossary

Quick glossary

Stored-program architecture
A design where program instructions are stored in memory so the CPU can fetch and execute them.
Von Neumann architecture
A stored-program model where instructions and data share the same memory and access path.
Harvard architecture
A model that separates instruction access from data access.
Von Neumann bottleneck
The slowdown caused when instructions and data compete for the same memory route.

Remember

Ask two comparison questions

  • Do instructions and data share memory access?
  • Does the system need simplicity or reduced contention?
  • Which model best fits the purpose of the device?
