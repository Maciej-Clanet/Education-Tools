Overview

What you should know by the end of this lesson

Once a system knows an error occurred, it still needs a strategy for getting usable data. Error correction methods either ask for the data again or carry enough extra information to recover without a retransmission.

Learning goals

  • Explain the difference between detection and correction.
  • Describe how automatic repeat request works.
  • Describe how forward error correction works.
  • Choose the right method for a scenario.

Quick memory hooks

  • ARQ: ask for the data again.
  • ACK: data received correctly.
  • NAK: data damaged or missing.
  • FEC: correct using added redundancy.

Why this matters

Some systems can afford to resend data, but others cannot. Live broadcasting, space communication, and one-way transmission often need correction to happen without waiting for another copy.

Core Idea

Error correction goes further than error detection

Detection tells the system that a problem exists. Correction aims to recover the correct data or replace the bad copy with a valid one.

Concept Main goal Typical response
Error detection Spot that the data changed Reject, warn, or request action
Error correction Recover usable correct data Retransmit or reconstruct

ARQ

Automatic repeat request fixes errors by retransmitting the data

ARQ relies on feedback. The receiver checks the data and then sends a message back. If the data is correct, it sends an acknowledgement. If the data is damaged or does not arrive properly, it requests another copy.

ACK data accepted NAK send it again Timeout no reply means retry

How the cycle works

  1. The sender transmits the data.
  2. The receiver checks it for errors.
  3. If the data is correct, the receiver sends an ACK.
  4. If the data is wrong, or if no reply arrives in time, the sender retransmits.

Where ARQ works well

ARQ is effective when two-way communication is available and there is enough time to request another copy, such as in many data networks and file transfers.

FEC

Forward error correction adds enough redundancy to recover without asking again

FEC sends extra check information along with the original data. That redundancy allows the receiver to correct certain errors immediately without needing a retransmission.

Main advantage No return message is needed, so correction can happen even on one-way or high-delay links.
Main trade-off Extra redundancy increases the amount of data sent and the processing needed.
Good fit Broadcast systems, satellites, and live media where retransmission would be slow or impossible.
Key idea The receiver can rebuild the correct value from the redundant information already provided.

Why FEC is useful for live or distant links

If you are streaming live media or sending data across a link with huge delay, waiting for a resend can be too slow. FEC trades extra data overhead for faster recovery at the receiving end.

Compare

Choose ARQ or FEC by asking whether retransmission is practical

The key scenario question is usually not "Which is better in general?" but "Which is better for this job?"

ARQ versus FEC

Method Best point Main weakness Typical fit
ARQ Can recover by resending the correct data Needs time and a return path Two-way file transfer or normal networking
FEC Corrects without waiting for a resend Needs extra redundant data Broadcast, live streams, long-delay links

Model answer shape

A stronger response sounds like this: "FEC is suitable because the receiver must correct errors immediately during a live one-way broadcast, so there is no practical opportunity to request the data again."

Exam Traps

Common mistakes with ARQ and FEC

These answers often lose marks when the timing or communication direction is ignored.

Thinking ARQ works without a return path

ARQ depends on acknowledgements, negative acknowledgements, or timeouts, so some form of feedback is essential.

  • ARQ needs two-way communication.
  • Without feedback, the sender cannot know to resend.

Fix: Mention ACK, NAK, or timeout when you describe ARQ.

Assuming FEC has no cost

FEC avoids retransmission delays, but it does so by sending extra redundant information.

  • More redundancy means more overhead.
  • That overhead can be worth it in the right scenario.

Fix: Explain both the benefit and the extra data cost.

Choosing ARQ for live one-way delivery by default

If retransmission would be too slow or impossible, ARQ is not the strongest answer.

  • Live broadcast often suits FEC better.
  • File transfer often suits ARQ better.

Fix: Match the method to the scenario rather than memorising one "best" choice.

4 marks Explain

Question 1

Explain how automatic repeat request can correct a damaged transmitted block, and why this method depends on a reply from the receiving system.

Show answer guide

A strong answer explains both the resend cycle and the need for feedback.

  • The sender transmits the data block.
  • The receiver checks whether it arrived correctly.
  • If it is correct, the receiver sends an acknowledgement.
  • If it is damaged, missing, or not acknowledged in time, the sender retransmits the block.
  • This depends on a reply path because the sender must be told whether a resend is needed.
5 marks Compare and justify

Question 2

A live sports stream is sent to many viewers at once. Compare ARQ and FEC, then justify which method is more suitable for this situation.

Show answer guide

The best answers focus on timing and retransmission practicality.

  • ARQ corrects by asking for another copy, which takes time and needs feedback.
  • FEC sends extra redundant information so the receiver can correct errors immediately.
  • Live streaming is sensitive to delay because viewers need the content straight away.
  • Requesting a resend for every viewer would be impractical and could interrupt the live stream.
  • FEC is therefore the stronger choice because it supports immediate recovery without waiting for retransmission.

Glossary

Quick glossary

ARQ
Automatic repeat request, which relies on retransmission.
ACK
An acknowledgement that data arrived correctly.
NAK
A negative acknowledgement requesting another copy.
FEC
Forward error correction using added redundancy.

Remember

Resend or rebuild

  • FEC corrects using extra information already sent.
  • The better choice depends on delay and feedback options.
