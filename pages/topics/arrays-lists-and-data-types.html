Overview

What you should know by the end of this lesson

Arrays and lists both store collections of related items, but they do not organise memory or access data in the same way. Exam questions often reward answers that explain not just what the structure is called, but why its behaviour suits a task.

Learning goals

  • Define arrays and lists accurately.
  • Explain indexed access and linked traversal.
  • Compare software use with hardware implications.
  • Choose the better structure for a given scenario.

Quick memory hooks

  • Array: same-type items, ordered by index.
  • List: linked items, easy to grow or shrink.
  • Index: the position number of an item.
  • Pointer: a stored reference to another item.

Where they appear

Arrays suit timetables, monthly sales figures, and image pixels. Lists suit playlists, task collections, and data that changes size frequently while a program is running.

Array

An array stores similar items in indexed positions

An array is an ordered collection where each item has an index. In many exam contexts, arrays are treated as storing items of the same data type in a predictable sequence, which makes direct access by position straightforward.

Indexed each position has a number Ordered values keep a fixed sequence Direct access read item 4 without scanning first
Why it works The program can calculate where an indexed value should be.
Typical software use Test scores, timetables, pixels, and any fixed-position data.
Typical hardware link Contiguous storage helps predictable addressing and efficient reads.
Key implication Inserting into the middle can mean shifting later values.

Worked example

If an array stores exam scores at indexes 0 to 4, the program can jump straight to index 3 when it needs the fourth score. That is why arrays are strong when position matters.

List

A list can grow and shrink more easily as values change

A list is also an ordered collection, but exam answers often focus on the idea that items can be linked together rather than stored in one continuous block. That makes lists useful when values are added or removed regularly.

Linked each item can point to the next Dynamic size can change during use Traversal reach later items by moving through the chain

Flexible size

A list is useful when the number of items is hard to predict in advance.

Typical software use

Playlists, active tasks, or collections that change during a session.

Typical hardware link

Lists may need extra memory for pointers and may not sit next to each other in RAM.

Worked example

If a music app keeps a linked list of queued songs, adding a new song between two existing songs can be done by changing the links. The app does not need to shift every later song into a new indexed slot first.

Data Types

Data types define what is stored and how it can be handled

In this part of Unit 2, arrays and lists are treated as data types used to organise collections of data. Good answers explain their features, how software uses them, and why their storage model matters to the hardware underneath.

In software

Developers choose a data type because of the operations the program needs most often, such as direct indexed access or frequent insertion and deletion.

In hardware

Arrays suit predictable memory addressing. Lists can reduce shifting but need extra references and may involve more memory jumps during traversal.

In exam answers

Do not stop at naming the structure. Link the task, the access pattern, and the implication for storage or performance.

Mini summary

A data type is not just a label. It shapes the valid structure of the data, the operations that make sense, and the cost of common actions such as reading, inserting, or removing values.

Comparison

Choose the right structure for the job

Arrays and lists can both store ordered values, but they reward different priorities. The best choice depends on whether fast indexed access or flexible updates matters more.

Feature Array List
Main organisation Indexed positions in order Linked items in sequence
Best for Direct access to a known position Collections that change size often
Memory pattern Usually contiguous and predictable Often non-contiguous with extra pointer storage
Adding in the middle Can require shifting later items Can be easier if the correct links are known
Main caution Less flexible when size changes a lot Slower to reach a later item directly

Choose an array when...

Position matters, size is reasonably stable, or fast direct lookup by index is the main goal.

Choose a list when...

The collection changes often and the program benefits from easier insertion or removal.

Exam reminder

Make the trade-off explicit. Structure choice is usually about access speed versus update flexibility.

Exam Traps

Common mistakes students make with arrays and lists

Many weaker answers name a structure without explaining the behaviour that makes it suitable. These traps come up often in written questions.

Treating a list like just "a list of things"

Students sometimes use the everyday meaning of list and forget the computer science idea of linked items and dynamic size.

  • An exam list answer should mention links or flexible size.
  • An everyday shopping list is not enough explanation by itself.

Fix: Explain how the items are organised, not just that there are several of them.

Forgetting that arrays are indexed

Some answers describe arrays only as "storing lots of data" and miss the reason they are useful.

  • Each value has a position number.
  • Programs can reach a chosen position directly.

Fix: Use the word index when you explain why an array fits the scenario.

Ignoring hardware implications

Stronger Unit 2 answers often connect the data structure to memory layout or storage overhead.

  • Arrays usually rely on predictable neighbouring storage.
  • Lists often need pointers and extra traversal steps.

Fix: Add one sentence about memory, addressing, or pointer overhead when the question invites it.

4 marks Explain

Question 1

Explain why an array is a suitable way to store the daily temperature for each day of one week.

Show answer guide

A strong answer explains both the software benefit and the storage implication.

  • There are a fixed number of values because the week has seven days.
  • Each temperature can be stored at a known index, such as day 0 to day 6.
  • The program can access a chosen day directly by index.
  • Arrays suit ordered, predictable storage where direct access matters.
5 marks Compare and justify

Question 2

A music app lets users keep inserting and removing songs in the middle of a current queue. Explain why a list may be more suitable than an array.

Show answer guide

The best answers compare both structures instead of only describing the chosen one.

  • A list can grow and shrink more flexibly while the app is running.
  • When songs are inserted or removed often, a list can avoid shifting many later items.
  • Linked items can be updated by changing references between nodes.
  • An array is stronger for direct indexed access, but that is less important here.
  • A list may need extra pointer storage, but its update behaviour better matches this scenario.

Glossary

Quick glossary

Array
An ordered collection of items stored by index.
List
A collection of linked items that can grow or shrink.
Index
The numbered position of an item in an array.
Pointer
A stored reference that links one item to another.

Remember

Think about access and updates

  • Arrays are strong when indexed access matters most.
  • Lists are strong when the collection changes often.
  • Hardware implications can strengthen a written answer.
