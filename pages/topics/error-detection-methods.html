Overview

What you should know by the end of this lesson

Error detection does not fix damaged data by itself. Its job is to warn the system that something changed during transmission so the data can be checked, rejected, or sent again.

Learning goals

  • Explain how parity schemes detect simple bit errors.
  • Describe checksum and repetition schemes clearly.
  • Explain what CRC is used for.
  • Compare the strengths and limits of different methods.

Quick memory hooks

  • Detection: spots a problem.
  • Parity: checks odd or even bit count.
  • Checksum: compares a calculated value.
  • CRC: strong patterned error checking.

Why this matters

If a network or storage system cannot tell that data changed, it may pass on corrupted information as if it were valid. Detection methods give systems a chance to notice faults before the data is trusted.

Parity

Parity schemes count the 1 bits to look for a mismatch

A parity bit is added so the total number of 1s becomes either even or odd, depending on the chosen scheme. When the data arrives, the system counts the 1s again. If the count no longer matches the expected parity, an error is detected.

Even parity total 1s should be even Odd parity total 1s should be odd Bit flip parity can reveal the change

Worked example

Suppose the data is 1011001. It contains four 1s, so with even parity the parity bit would be 0 because the total is already even. If one bit changes during transmission and the receiver sees an odd number of 1s instead, the system knows corruption happened.

What parity can and cannot do

Strength Why it helps Limitation
Simple Quick to add and check Only gives a warning, not a repair
Useful for single-bit problems A single flipped bit changes the parity count Some multi-bit errors can slip through
Low overhead Needs very little extra data Less robust than stronger detection methods

Checksums And Repetition

Checksum and repetition schemes compare extra evidence with the original data

A checksum is a value calculated from the data before transmission. The receiver calculates the checksum again and compares it with the transmitted one. If they differ, the data is assumed to have changed.

Checksum A calculated summary value sent with the data for comparison at the destination.
Why it helps The whole data block affects the check value, so a mismatch is often easy to spot.
Repetition scheme The same data is sent more than once so the copies can be compared.
Main trade-off Repetition increases reliability but also increases bandwidth use because more copies travel.

When repetition helps

If a short control message is very important, sending it more than once can reduce the chance that a single damaged copy is accepted. The downside is obvious: repeating data takes longer and uses more network capacity.

CRC

Cyclic redundancy check is a stronger way to detect transmission changes

CRC is widely used because it is strong at spotting common error patterns in blocks of transmitted data. The sender calculates a check value from the message and the receiver repeats the process to see whether the result still matches.

Calculated check value travels with the data Recalculated on arrival for comparison Good for blocks and patterned errors

Why CRC is popular

CRC is more sophisticated than simple parity and is strong at spotting burst errors, where several nearby bits change. That makes it useful in many networking and storage contexts where a stronger check is needed without fully correcting the data.

Compare

Choose a detection method by balancing simplicity and strength

Stronger detection usually means more processing or more extra data, but simple methods can miss important faults.

Method comparison

Method Best point Main weakness Typical use idea
Parity Very simple and low overhead Limited error coverage Basic checking for small transmissions
Checksum Good general block comparison Still only detects, not corrects Files or packet blocks
Repetition Easy to compare multiple copies Uses extra bandwidth Important short messages
CRC Strong detection for many error patterns More complex than parity Networks and storage systems

Exam Traps

Common mistakes with error detection

These questions often hinge on the difference between noticing a problem and actually repairing it.

Confusing detection with correction

A detection method tells the system something went wrong. It does not automatically rebuild the correct data.

  • Detection spots the fault.
  • Correction fixes or replaces the damaged data.

Fix: Use the phrase "detects an error" rather than "fixes an error" for parity, checksum, and CRC.

Assuming parity catches every mistake

Parity is useful, but it is limited and can miss some multiple-bit changes.

  • It is strongest for simple odd or even mismatches.
  • It is not a complete protection system.

Fix: Mention parity as a simple check, not a universal solution.

Ignoring the bandwidth cost of repetition

Sending the same data again can increase confidence, but it also increases transmission overhead.

  • More copies mean more data sent.
  • That can reduce efficiency.

Fix: Always mention the trade-off.

4 marks Explain

Question 1

Explain how an even parity scheme can detect a simple transmission error, and why it does not tell the receiver the correct original bit pattern.

Show answer guide

A strong answer explains both the check and the limit.

  • Even parity adds a parity bit so the total number of 1s should be even.
  • The receiver counts the 1s again when the data arrives.
  • If the total is no longer even, the system detects that an error occurred.
  • Parity only signals that something changed. It does not identify the exact original data to restore.
5 marks Compare and justify

Question 2

A company sends large file blocks across a busy network. Compare checksum, repetition schemes, and CRC, then justify which detection method is likely to be strongest for this situation.

Show answer guide

The best answers compare reliability with overhead.

  • Checksum provides a calculated comparison value for the whole block.
  • Repetition sends extra copies but uses more bandwidth.
  • CRC is strong at spotting many error patterns in block data.
  • For large busy transmissions, repetition may be less efficient because of the extra data sent.
  • CRC is usually the strongest justified choice because it gives strong detection without simply duplicating the full data block.

Glossary

Quick glossary

Parity bit
An extra bit used to support odd or even parity checking.
Checksum
A calculated value used to compare data integrity.
Repetition scheme
A method that sends multiple copies for comparison.
CRC
A cyclic redundancy check used for stronger detection.

Remember

Spot, then respond

  • Error detection warns that something changed.
  • Different methods trade simplicity against strength.
  • Detection alone does not rebuild the lost data.
